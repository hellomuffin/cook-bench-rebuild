Lesson · Controlling the cook with a VLM

The cook speaks manipulation, not magic.

A streaming VLM controls the cook through small physical commands — "go to the board", "pick up", "chop", "put down". Crucially, no command does the task for it: there is no "make the soup", no "chop the potato" that fetches and dices in one breath. The model must compose a chain and time it against what it sees. Build one yourself in the demo below.

The design rule

Abstract the motor control — never the task

We hide tedious low-level motor control (pathfinding, turning to face a tile) but we deliberately keep every bit of task functionality in the VLM's hands. A command is a single physical act on one object; it is never a sub-goal.

⌨️ Low level — what the engine runs

6 buttons, one per tick; every interaction is the same button.

▲▼◀▶● interact… stay

Too fiddly for a VLM to drive frame-by-frame.

🤖 Manipulation — what the VLM gets ✅

One mover + atomic object manipulations. Each acts on the tile the cook already faces.

"go to the cutting board"
"put down" · "chop" · "pick up"

No task logic — the model supplies the plan.

What we deliberately do NOT give: no chop the potato cook the soup serve onion soup. Each of those bundles fetching, ordering, timing and plating — i.e. the functionality. Hand those to the VLM and the task is trivial. Withhold them and it has to actually understand the kitchen.
The vocabulary

Two kinds of command

① Locomotion — the only thing that moves

Exactly one command moves the cook, and it is the only command that names a target (and therefore carries the grounding qualifiers below). Everything else acts on whatever the cook is currently facing.

🚶go to <object>
Pathfind next to an object and turn to face it. Abstracts away every footstep + the orient-first turn.
⤷ BFS path · face the tile

② Manipulation — one object operation, where you stand

Each acts on the tile the cook faces and touches one object. It does not walk, fetch, sequence, or plate. If the cook isn't in position, the command simply fails ("you're not at a board") — so the model must go to first.

Grounding (only on "go to")

Which object do you mean?

Only go to picks an object, so that's the only place ambiguity lives. When several candidates exist ("the pot" with two pots), the VLM attaches a qualifier that injects spatial / relational meaning. Manipulations need no target — they act on the faced tile.

QualifierSay it likeResolves by
content"go to the pot cooking rice"what's inside / its job

Proof — pure-vision streaming, no state leak

Watch Gemini play it, frame by frame

Honest input: the game runs continuously and the model sees only the real rendered 3D frames (no labels, no status text, no coordinates, no "holding"). We adapt an offline model into a streaming controller with the sliding-window policy (the strong inference-time strategy from arXiv:2604.07634 §4.1): about once per second we send the model the last 5 frames (oldest→newest, so it can see motion) plus its last 5 responses, and it answers with an empty action (no-op) or one command — judging from the pixels whether its last command has finished. …and the real Gemini Live API (audio-native, true streaming session)

The two above adapt an offline model. This one uses the actual Gemini Live bidi API: a persistent session with its own internal memory, frames streamed in, and the model's reply spoken as audio → transcribed to text. Gemini Live is not proactive, so we ping it every frame ("next action?") — the system prompt holds all the context once, up front. The HUD's command is the model's spoken sentence, transcribed.

The difficulty ladder — how far streaming control gets

A gradient, not a single cliff

The onion-soup task above is a single decision: look, identify, scoop. To probe the limits of streaming control we add a ladder of ordered, multi-step tasks — each must be assembled on the plate in an exact sequence, and the engine rejects a wrong-order plate. Burning is disabled, so these are pure multi-step reasoning tests, not cook-timer races. All three are verified feasible — the scripted BFS bot completes each.

Result. The strongest model on this key (gemini-3.1-pro-preview, sliding-window, pure RGB) degrades smoothly with horizon: it solves the instance task (+21), and on the 3-component Cheeseburger it usually delivers (+34, ≈3 of 4 runs) — recovering from a handful of mis-timed commands — but it fails the longer Fried Rice and Deluxe Burger (3 tools / 4 ordered components). The audio-native Gemini Live model is weaker still and fails from the Cheeseburger up. The recurring failure: the model gathers, cooks and chops the right ingredients but loses track of what it has already added from pixels alone and violates the required order — the long-horizon, state-tracking gap a streaming-interaction benchmark exists to surface.

Which commands get dropped — and whose fault?

A manipulation only fires when the cook is standing at and facing the right tile; the sliding-window agent drops one issued out of position (the Gemini Live agent instead queues it). Instrumenting three Cheeseburger runs (gemini-3.1-pro-preview), 72 commands were issued, 62 executed, 10 were dropped as illegal — a representative run drops 4 then recovers. Every dropped command and its engine reason:

×commandengine reason it was rejected root cause
3put down"can't put that down here — face a counter or an open pan" (still walking to the pan)VLM timing — issued a beat before arrival
1scoop"nothing ready to scoop here — face a done station / chopped board / source" (not yet arrived)VLM timing — issued a beat before arrival
2turn on"nothing to start — the station is empty"VLM cascade — assumed the dropped put down had worked
2pick up"your hands are full"VLM state-tracking — it was already holding the meat (visible in-frame)
2scoop"hold a clean plate to scoop onto"VLM sequencing — tried to plate before fetching a plate
Spec ambiguity vs VLM fault. 0 of 10 drops come from an unclear task or action spec — every rejection maps to a precondition stated verbatim in the system prompt ("a manipulation only works while standing at and FACING the right tile — go to it first", "HOLD a plate and scoop", "say continue until you SEE it has arrived") and the engine's error messages are precise. The split is:
  • ≈40% timing slips (the put down / scoop issued one decision early). These are soft: the model judged "I've arrived" a beat too soon. Under the Live agent's intent-queuing they would simply fire on arrival instead of being dropped — so they're as much an agent-policy choice (drop vs queue) as a model error.
  • ≈60% genuine VLM reasoning errors — a pick up with full hands or a scoop with no plate is a state-tracking / planning mistake the picture already disproves; the turn on failures are cascades from acting as if a dropped command had succeeded.
Net: the spec is clear; the drops are the model's, but the sliding-window drop policy converts forgivable timing slips into wasted turns (and occasional cascades). It still usually recovers on the Cheeseburger; on the longer tasks the same slips compound past recovery.

Task specifications — recipe, ground-truth steps, mechanics

Each task gives the model the recipe text below (appended to the system prompt). Burning is disabled (pot/pan/oven_burn_time = 10⁹) so a cooked item can wait — the test is the reasoning, not a timer race. Engine constants used below: pot cap 3 / cook 20t · pan cap 2 / cook 15t · oven cap 3 / cook 28t · cutting board 4 chop-interactions · a cooking station starts when full, or by an empty-handed turn on · to plate cooked/chopped food you hold a plate and scoop at the done station.

🍔 hard · Cheeseburger  layout: burger_bar 3 components, ordered reward +34
Recipe prompt given to the model: "RECIPE — Cheeseburger (build the plate in THIS EXACT ORDER): (1) a BUN (raw, from the bun source), then (2) a FRIED PATTY — cook a piece of meat in the PAN first, then (3) CHEESE (raw). To put each item on the plate, HOLD the plate and 'scoop' at the bun source / the done pan / the cheese source. Add them bun -> patty -> cheese, then serve." Ground-truth steps:
  1. go to the meat source → pick up raw meat
  2. go to a pan → put down the meat → turn on (fries in 15t; can wait)
  3. go to the plates → pick up a clean plate
  4. go to the bun source → scoop → plate = [bun·raw]
  5. go to the pan (patty done) → scoop → plate = [bun, meat·cooked]
  6. go to the cheese source → scoop → plate = [bun, meat·cooked, cheese·raw]
  7. go to the serving pass → serve → matches Cheeseburger → +34

Recipe prompt given to the model: "RECIPE — Fried Rice (build the plate in THIS EXACT ORDER, three different cooking tools): (1) COOKED RICE — cook rice in a POT; (2) a FRIED EGG — cook an egg in the PAN; (3) CHOPPED ONION — chop an onion on the BOARD. Then hold a plate and scoop them onto it in the order rice -> egg -> onion, and serve. (There are two pots and two boards — any will do.)" Ground-truth steps:
  1. rice source → pick up; a pot → put down → turn on (cooks 20t)
  2. egg source → pick up; a pan → put down → turn on (fries 15t)
  3. onion source → pick up; a cutting board → put down → chop ×4
  4. go to the plates → pick up a clean plate
  5. pot → scoop → [rice·cooked] · pan → scoop → [rice, egg·cooked] · board → scoop → [rice, egg, onion·chopped]
  6. go to the serving pass → serve → matches Fried Rice → +44
🍔 hardest · Deluxe Burger  layout: diner 4 components, ordered, ONE board reward +46
Recipe prompt given to the model: "RECIPE — Deluxe Burger (build the plate in THIS EXACT ORDER, four components): (1) BUN (raw); (2) FRIED PATTY (cook meat in the PAN); (3) CHOPPED LETTUCE (chop on the BOARD); (4) CHOPPED TOMATO (chop on the BOARD). There is ONE board, so chop lettuce first, scoop it onto the plate, then chop the tomato. Add bun -> patty -> lettuce -> tomato in order, then serve." Ground-truth steps (the single board forces the interleaving):
  1. meat → pick up; pan → put down → turn on (patty cooks 15t)
  2. lettuce → pick up; board → put down → chop ×4
  3. the plates → pick up a clean plate
  4. bun source → scoop → [bun] · pan → scoop → [bun, meat·cooked] · board → scoop → [bun, meat, lettuce·chopped] (board now free)
  5. put down the plate on a counter; tomato → pick up; board → put down → chop ×4
  6. pick up the plate again; board → scoop → [bun, meat, lettuce, tomato·chopped]
  7. go to the serving pass → serve → matches Deluxe Burger → +46

How success is evaluated

There is no "deliver" verb — a dish is judged only at the moment you serve while standing at the serving pass (KitchenGame._interact_serving):

  1. You must be holding a clean, non-empty plate (a dirty plate or empty hands does nothing).
  2. RecipeBook.match_plate checks the plate against the menu: its multiset of (ingredient, state) must equal a recipe's, and because these are ordered recipes it additionally requires the plate's addition sequence (plate.content_seq) to exactly equal the recipe's component order. Any burnt item disqualifies it (not possible here — burning is off).
  3. If it matches an open order, you get the recipe's base_reward (34 / 44 / 46), deliveries += 1, and the plate is returned. That is the only success signal we score.
  4. Any mismatch — wrong items, missing/extra component, or right items in the wrong order — is a failed delivery: the food is discarded, the plate is dirtied, and you take invalid_serve_penalty = −3. (This is exactly how the cheeseburger run scored −1→fail: it plated bun → cheese → patty, the wrong order, so the ordered recipe rejected it.)

How we prompt Gemini Live

Gemini Live is a stateful, audio-native bidi session and is not proactive — it only speaks when prompted. So all the context goes in the system instruction once, then every frame we stream the rendered RGB image and send a tiny text ping; the model speaks one short command (or "continue"), which we transcribe (output_audio_transcription) and parse. The session keeps its own internal memory of past frames + replies (context_window_compression = sliding_window), so we never resend history.

① Initial system instruction (sent once at session open; the recipe text above is appended verbatim):

You are a STREAMING controller for one cook in a 3D top-down cooking game. You watch a live video and
I will ask you, frame by frame, for the next action. Read everything from the picture: where the cook is (the
blue chef), which way it faces, what it holds (a white plate, with food on it once filled), what is inside each
pot/pan, and whether food is still cooking vs done.

EACH TIME I ask, reply by SAYING exactly ONE short thing and nothing else:
- say "continue" = NO-OP / no interrupt: keep doing the current thing (use this while the cook is still walking,
  or while food is still cooking; you judge from the video whether the last command has finished).
- OR say one command.

INSTANCE / SPATIAL GROUNDING — IMPORTANT: when several of an object exist (two pots) they may hold DIFFERENT
food. LOOK to see which has what you need (yellow onions vs red tomatoes) and name THAT one: "the pot with the
onions", "the left pot", "the right pot". Choosing the wrong instance ruins the dish.

COMMANDS: "go to <object>" (the only move; + the instance qualifiers; objects: an ingredient source, a pot /
pan / cutting board / sink, "the plates", "the serving pass") · "pick up" · "put down" · "chop" · "turn on" ·
"scoop" (hold a plate, add the faced station's food onto it) · "serve".
RULES: a manipulation only works while standing at and FACING the right tile, else nothing happens — go to it
first. To plate cooked food, HOLD a plate and "scoop" (don't "pick up"). The cook moves one tile per frame and
takes several frames to arrive — say "continue" until you SEE it has clearly arrived. Keep replies to a few words.
IMPORTANT: only say "continue" while the cook is actively walking or food is cooking. If the cook is standing
still with nothing in progress, do NOT say continue — say the NEXT command to make progress toward the dish
(plan: get a plate -> go to the onion pot -> scoop -> go to the serving pass -> serve).

Offline sliding-window agent: the recipe is given the same way but, because each generate_content call is stateless, it is prepended to every per-frame message (f"{recipe}\n\nYour last 5 actions…" + the last 5 frames) rather than set once as a session system instruction.

② Per-frame streaming prompt (every tick, after streaming the RGB frame via send_realtime_input(video=…)):

Next action?

That two-word ping is the whole per-frame message — everything else (the rules, the recipe, the running history of frames and its own past replies) is already in the system instruction and the session's internal memory. We read back the spoken reply, map "continue"/"wait"/… to a no-op, otherwise parse the one command; a manipulation said before the cook is in position is queued and auto-fires on arrival (intent-queuing).
