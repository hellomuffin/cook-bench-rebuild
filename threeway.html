Which agent, which input?

Three architectures — a reactive controller (command each step), plan-act simple (flat plan then execute), and plan-act structured (PWR-style revisable plan + visual cues) — each across five input settings. All under one fair config: single standing order, cap 300 frames, no early give-up. gemini-3.1-pro, 9 tasks × 3 reps per cell. Below: the grid, then a comprehensive error analysis per agent.

Hard Pizza (3 simultaneous cooks) is 0% on every RGB condition — including the full feedback ladder — but 6/8 with the labelled state map. Given a clean readout of what's cooked / on the plate / where things are, the reactive model handles the 3-cook Pizza; it simply can't read that state from pixels. So the earlier "can't plan Pizza" is really can't perceive the Pizza state.

Cross-cutting findings

1. Illegal-command rate is the master signal. Across all agents it moves inversely with solve rate — e.g. controller 37%→4% (basic→state-map), plan-act simple 48%→2% (basic→agent++). Bad inputs → the model issues commands the engine rejects → it derails.
2. "Plan-correct but still failed" proves basic's 0% is execution, not planning. Plan-act's self-made plans are ~60% correct (they deliver under the robust harness), yet in basic every correct plan fails when the VLM executes it live — with feedback that collapses to ~0 failed. Feedback is what makes live execution work.
3. The state map beats the text feedback ladder for the reactive controller (81% vs 48% at agent+), and is the only input that cracks hard Pizza — a strong vote for giving the model structured perception over more text.
4. Structured (PWR-style) re-planning pays off, but only with feedback — it revises its plan several times per run and reaches 100% at agent+++, vs simple's 78%.
