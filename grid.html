Procedurally scaling task complexity

One axis at a time

Every task here is procedurally generated (cooksim.core.generator) and rated on a formal 3-number complexity vector: n_ops = minimal number of high-level VLM commands · path = tiles the cook must walk along the optimal plan · map = recipe-independent kitchen difficulty (grounding ambiguity + visual clutter/size + navigation tortuosity). Below we hold one axis fixed and scale the others, so you can see exactly what each axis means. Each card shows the quantized levels and the full ground-truth plan in the VLM action space (go to / pick up / put down / turn on / chop / scoop / serve).

① n_ops × path, map held ≈ 5.75

Rows scale n_ops (easy→hard), columns scale path (easy→hard); map complexity is held constant (≈5.75) across all cells. The medium / medium cell is our Cheeseburger.

Why one cell is empty. The three axes are not fully independent: a hard-n_ops task has many components, so the cook must visit many sources and stations — its minimum route is already “medium”. A hard task therefore cannot also be easy-path. That geometric coupling is itself a property of the complexity space.

② map complexity alone — easy n_ops, path free

The same easy recipe (Caprese) in three kitchens of growing map complexity. Only the kitchen changes — here it scales by sheer size (more floor to scan & cover); ambiguity and tortuosity are the other dials.
