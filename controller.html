Streaming-VLM controller — results

Can a streaming VLM drive the cook?

The VLM is the controller: each ~second it sees only the rendered RGB and issues one mid-level command; success = the dish is delivered. gemini-3.1-pro-preview (sliding-window), 9 procedurally-generated tasks (n_ops×map), 5 input conditions, 3 reps = 135 runs. Pure-pixel input; every decision logged. Overall solve rate: 29/135 (21%).

1 · Effect of input condition

basic = RGB only · graph_state = labelled state map · agent+ = +per-step actions · agent++ = +location · agent+++ = +plate text.

State helps, monotonically — RGB-only 7% → agent+++ 33%; the state map alone nearly triples pure RGB. A lot of the difficulty is perception.

3 · Error analysis — how the 106 failures fail

It is almost never a timeout. The model reaches a serve but the plate is wrong:

  • served an invalid plate — 103/106 of failures (97%)
  • stuck / aborted — 3/106 of failures (3%)

Reconstructable served plates are overwhelmingly incomplete (a required component missing) — the model commits to a serve too early. Wrong-order is also penalised but is hard to see in the end-state (the plate renders as a pile), so it must be caught from the trajectory.
